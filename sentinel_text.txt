Introduction 1.1 What Is SENTINEL? SENTINEL is an AI-powered Incident Response and Threat Intelligence platform built specifically for small businesses — the organizations that are most frequently targeted by cybercriminals but least equipped to defend themselves. A small clinic in Lahore, a school in Karachi, a boutique fintech in Islamabad — none of them can afford Splunk or CrowdStrike. But all of them face ransomware, phishing, account takeovers, and data breaches every day. SENTINEL gives them a fighting chance.  Core value: A small business owner or their IT person should be able to log an incident, get a step-by-step response playbook, check if their IPs/emails are malicious, and generate a management report How SENTINEL Works — Technical Deep Dive 4.1 Threat Intelligence Pipeline When a user submits an indicator (e.g. IP address 185.220.101.45), this is what happens under the hood:  Step Phase What Happens Tool/Method T1 Input validation Detect indicator type (IP/domain/hash/email/URL) using regex patterns Python regex + validators library T2 Cache check Check SQLite cache — was this indicator checked in last 24h? SQLite lookup T3 Parallel fetch Query ALL threat feeds simultaneously using asyncio Python asyncio + httpx T4 AbuseIPDB GET /api/v2/check?ipAddress={ip}&maxAgeInDays=90 AbuseIPDB API T5 AlienVault OTX GET /api/v1/indicators/IPv4/{ip}/general OTX API T6 VirusTotal GET /api/v3/ip_addresses/{ip} VT API v3 T7 ThreatFox POST /api/v1/ with {query: 'search_ioc', search_term: ip} ThreatFox API T8 IPinfo GET /{ip}/json — geolocation + ASN + org IPinfo API T9 Aggregate Combine all responses into structured dict Python dict merge T10 AI Verdict Send aggregated results to Claude — generate verdict + explanation Claude API T11 Cache store Save result to SQLite with 24h TTL SQLite insert T12 Return Send verdict + all feed results to frontend FastAPI JSON response  Why parallel fetching? If we checked feeds one by one: 10 feeds x 3 seconds = 30 seconds. With asyncio parallel: all 10 simultaneously = ~4 seconds. This is critical for user experience.  4.2 AI Classification + Playbook Generation When an incident is logged, AI does two things: classifies the incident, then generates a custom playbook.  Classification Prompt: SYSTEM: You are a cybersecurity incident response expert. Analyze this incident report and return JSON with: {   "incident_type": "ransomware|phishing|account_takeover|data_breach|insider|ddos|social_eng|lost_device|vendor_breach|other",   "severity": "critical|high|medium|low",   "severity_reason": "one sentence why this severity",   "affected_data_types": ["PII","financial","credentials","IP","health"],   "iocs_extracted": ["any IPs, domains, hashes mentioned in description"],   "peca_reporting_required": true|false,   "peca_reason": "why or why not" } INCIDENT: {incident_description}  Playbook Generation Prompt: SYSTEM: You are a senior IR consultant creating a response playbook. The audience is a small business with NO dedicated IT security staff. Be specific, practical, and use plain language. Return JSON: {phases: [{phase_name, steps: [{step, owner, time_estimate, critical: bool}]}],  immediate_actions: [top 3 things to do RIGHT NOW],  notify_list: [who must be told and why],  pakistan_specific: [FIA/PECA/SBP/PTA specific steps if applicable]} INCIDENT TYPE: {type} | SEVERITY: {severity} | DETAILS: {description}  4.3 Log Analysis Pipeline Step Phase What Happens Tool/Method L1 Upload User uploads log file (max 50MB) React file upload + FastAPI L2 Detect type Identify log format from first 20 lines Regex pattern matching L3 Parse Convert raw log to structured list of entries Python parsers per log type L4 Extract IPs Pull all IP addresses from parsed logs Regex + ipaddress library L5 TI check Bulk check extracted IPs against threat feeds TI Engine (Module 1) L6 Pattern detect Apply detection rules to parsed entries Python rule engine L7 AI analysis Send suspicious entries + stats to Claude Claude API L8 Plain English AI generates summary of what it found Claude response L9 Link to IR User can create incident from any finding Frontend action button  Detection Rules (hardcoded, fast): Brute force: >10 failed logins from same IP within 5 minutes Off-hours: successful login between 11pm-5am local time Large transfer: outbound > 100MB to external IP New admin: event ID 4720 (Windows) or useradd in Linux auth log Impossible travel: same user, 2 different countries, less than 2 hours apart Known bad IP: any IP in logs that TI feeds flag as malicious  4.4 IOC Auto-Extraction from Text When an incident is described in plain text, AI extracts all IOCs automatically:  Input: 'We received a phishing email from support@micros0ft-verify.com with a link to http://185.220.101.45/login.php. The file attached was named invoice.exe with hash a1b2c3d4e5f6...'  AI Extracts: {   "emails": ["support@micros0ft-verify.com"],   "ips": ["185.220.101.45"],   "urls": ["http://185.220.101.45/login.php"],   "file_names": ["invoice.exe"],   "hashes": ["a1b2c3d4e5f6..."],   "domains": ["micros0ft-verify.com"] }  All automatically queued for TI feed checking.  4.5 Technology Stack  Backend Python 3.11 + FastAPI + asyncio for parallel TI feed queries Frontend React 18 + TypeScript + Tailwind CSS + Recharts (charts) Database SQLite (portable, zero setup) with SQLAlchemy ORM AI Engine Anthropic Claude API (claude-sonnet-4-6) TI Feeds AbuseIPDB, AlienVault OTX, VirusTotal, ThreatFox, URLhaus, IPinfo, MalwareBazaar Log Parsing Python-evtx (Windows logs), custom parsers for Linux/Apache/Nginx Report Generation ReportLab (PDF) + python-docx (Word) STIX Export stix2 Python library for IOC export Async HTTP httpx with asyncio for parallel API calls Caching SQLite-based TTL cache for TI results PART 3 — CLAUDE CODE SPRINT PROMPTS HOW TO USE: Open Claude Code.
